1. indentation is always 2 spaces.
2. index names must reflect their purpose, not just what they are (eg: search_by_street_id, not idx_house_numbers_street_id).
3. static sql strings must be declared as `const` at module level with a `SQL_` prefix (e.g. `SQL_INSERT`, `SQL_CREATE_TABLE`), not inlined in function bodies. multi-line sql must be indented with 2 spaces, each clause on its own line, with a leading and trailing newline inside the string literal. columns and values in insert statements must be listed one per line.
4. database ids must use unsigned integer types, sized as small as possible (e.g. `u8` for admin levels 1–30, `u32` for osm ids if range allows, `u64` only when necessary).
5. json columns must be stored as jsonb: use `jsonb(?n)` in insert/update statements, never plain `?n`. when reading a jsonb column back into rust via `fromsql`, the select must use `json(col)` to convert to text first — `fromsql` implementations use `value.as_str()` and will fail on raw jsonb blobs.
6. tasks are tracked in `tasks.md` at the project root. organize tasks under existing sections or create a new section if none fits. new tasks always go to the end of their section (queue, not stack). before executing a task from `tasks.md`, weigh whether it is actually necessary and whether it is redundant with existing functionality — surface the doubt and confirm before implementing. tasks are a backlog of ideas, not a queue of guaranteed work. every task in `tasks.md` must start with one of the action verbs `ADICIONAR`, `REMOVER`, or `MODIFICAR` and must end with a purpose clause starting with `a fim de` (e.g. `a fim de otimizar a performance`, `a fim de diminuir o tempo de extração`, `a fim de evitar regressões`). when adding or rewriting a task, enforce this shape; when grooming the backlog, rewrite legacy items in place to match.
7. sql functions must be written in uppercase (e.g. `JSON_EXTRACT`, `COALESCE`, `CAST`, `COUNT`).
8. `src/extract/admin_levels/readme.md` documents the default inclusion rules per level. keep it in sync with the extraction logic in source — any change to filters in `level_*.rs` must be reflected in the readme.
9. test names (file names and scenario names) must always be written in english, regardless of the language used elsewhere in the codebase.
10. dependencies must be added via `cargo add` and removed via `cargo remove`. never edit the `[dependencies]` section of `Cargo.toml` by hand.
